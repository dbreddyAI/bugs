Bug Triaging

Lucie Klimosova


Large software projects can produce hundreds of bug reports every day.
Many of these reports contain incomplete, incorrect, or redundant information.
Bug reporters and developers would benefit from a "smarter" bug reporting system.

Data

The Eclipse and Mozilla defect tracking dataset... [4]

Mozilla subset (mostly Firefox, Thunderbird, Mozilla projects).
Almost 400,000 bug reports resolved between 1998 and 2013.
Additional information obtained from Bugzilla API - full description, id of duplicate.

"Assigning severity to bug reports is highly dependent on the expertise of the bug reporters. Novice bug reporter might find it difficult to decide the right severity level."

Information retrieval for fine-grained bug severity prediction... [1]

"Correct prioritization of bugs helps in bug fix scheduling/assignment and resource allocation. Failure of this will result in delay of resolving important bugs."

Predicting the priority of a reported bug... [2]

"Users can, and often do, submit reports describing the same problem. The result is an added workload on the software maintenance engineers who have to ensure that each report is assigned a correct status."

Automated duplicate bug report classification... [3]

Insights

There are many areas for improvement.

72% of severity normal, 8% changes over the lifetime.
98% of priority empty, only 9% filled in over the lifetime.
Similar change rate applies to product, component and version.
The bug is assigned to many developers during the lifetime.

26% bug reports end up being closed as duplicates.

I have focused on predicting severity and priority and detecting duplicates.

Modeling

Default values (normal severity, empty priority) treated as unlabeled.

Severity

TfIdf on short descritption
TfIdf on long description
Gradient Boosting on other features
Linear stacking of resutls

Priority

TfIdf on short descritption
TfIdf on long description
Gradient Boosting on other features
Linear stacking of results

Duplicates

Problem on pairs of observations
Return duplicate high in the search
Similarity of description and product
(Experimented with LSH, doc2vec)

Results

Better than or comparable with previously published researches. [1][5][6]

Severity

  • accuracy: 68%
  • off by 1 accuracy: 93%

Priority

  • accuracy: 58%
  • off by 1 accuracy: 87%

Duplicates

  • top 3: 10%
  • top 10: 15%
